# 空窍容量与压力正式设计（v1）

## 1. 目的

这份文档用于把“空窍容量 / 压力 / 高转蛊压制”从一句话想法，升级为后续可以直接实现的正式规则。

目标不是再发明一套新系统，而是**把现有空窍基础设施补齐玩法真规则**：

- 空窍格子数不再只看真元，而是改为**资质决定基础容量，修为决定额外扩容**。
- 蛊虫放进空窍后，不只是“占一个格子”，还会持续占用**压力预算**。
- 低转空窍强行驾驭高转蛊时，会产生明显的**跨转数压制成本**。
- 被动常驻、主动轮盘、杀招施放，都必须吃压力预算，而不是白嫖运行。

---

## 2. 当前代码现状（必须承认的现实）

目前仓库里，空窍框架已经很完整，但玩法规则仍然偏薄：

### 2.1 已有的骨架

- `KongqiaoCapacityService`
  - 当前会根据 `ZhenYuanHelper.getMaxAmount(entity)` 把空窍容量同步到动态行数。
  - 现状是：**真元 = 容量**，尚未体现资质差异。
- `KongqiaoData`
  - 当前只承担空窍背包、攻击背包、喂食背包和 dirty 同步标记。
  - 现状是：**没有压力状态**。
- `GuRunningService`
  - 当前每 tick / 每秒遍历空窍中已解锁用途并执行效果。
  - 现状是：**还没有被动运行成本校验**。
- `ShazhaoActiveService`
  - 当前会检查杀招是否解锁、数据是否存在、需求蛊虫是否齐全。
  - 现状是：**还没有主动压力爆发 / 超压失败规则**。
- `TweakConfig`
  - 当前只记录“被动开关”和“轮盘技能列表”。
  - 现状是：**没有运行负荷、技能保留负担、超压锁定状态**。

### 2.2 现状问题

如果继续维持现在的结构，会产生几个明显问题：

1. 空窍更像“第二个背包”，不像修仙器官。
2. 高转蛊与低转蛊的驾驭成本差异不明显。
3. 被动越多越赚，没有持续代价。
4. 杀招激活只看解锁与材料，不看空窍是否撑得住。
5. 资质只在升仙准备中出现，没有真正反映到长期空窍运营。

---

## 3. 设计原则

### 3.1 容量与压力必须分离

- **容量（Slots）**：决定你能“装多少只蛊”。
- **压力（Pressure）**：决定你能“稳不稳地养、开不开得起、能不能持续运转”。

两者不能混成一个数字。

### 3.2 资质决定先天底盘，修为决定后天扩张

- 资质高的人，空窍先天底子更大，基础格子更多。
- 修为、真元、空窍发育，会继续提供额外扩容。

### 3.3 高转蛊不是简单“能不能放”，而是“放进去要付出多大代价”

低转空窍强装高转蛊，不应该一刀切禁止。

更符合蛊真人味道的做法是：

- **能放**，但要吃巨大压力；
- 压力过高时，被动先失稳，主动后失控；
- 再往上会出现反噬、封槽、掉负担，而不是无事发生。

### 3.4 服务端必须是权威

所有与压力有关的规则——包括：

- 当前压力值
- 被动负担
- 主动爆发负担
- 超压等级
- 封槽 / 失稳 / 激活失败

都必须由服务端判定。客户端只负责展示。

## 4. 容量模型（Slots）

## 4.1 总体思路

后续 `KongqiaoCapacityService` 不再把“最大真元”直接视为“总容量”，而是拆成：

`总行数 = 资质基础行数 + 修为追加行数`

其中：

- **资质基础行数**：体现先天资质差异。
- **修为追加行数**：复用当前基于真元的扩容思路。

## 4.2 资质档位与基础行数

> 注意：当前代码里能直接读到的只是 `AptitudeResourceState`（完整 / 缺项 / 全缺），这还不是一个可直接用于空窍长期玩法的数值资质源。实现时必须新增一个**明确的资质桥接层**，不能把 `AptitudeResourceState` 当成最终档位使用。

建议新增：

- `KongqiaoAptitudeTier`
- `KongqiaoAptitudeResolver` / `KongqiaoAptitudeBridge`

建议的基础档位如下：

| 资质档位 | 语义 | 基础行数 | 基础格子 |
|---|---|---:|---:|
| 残缺 | 勉强成窍，极难负担多蛊 | 1 | 9 |
| 下等 | 底盘偏小 | 2 | 18 |
| 中等 | 正常可玩基线 | 3 | 27 |
| 上等 | 明显优于常人 | 4 | 36 |
| 绝品 | 顶级底盘 | 5 | 45 |

## 4.3 修为追加行数

修为追加行数继续沿用当前 `KongqiaoCapacityService` 的思路，但语义要改成：

- 它不再代表“总容量”；
- 它只代表“在当前资质底盘之上的后天扩张”。

建议：

- 继续复用 `KongqiaoConstants.FULL_CAPACITY_ZHENYUAN = 1000.0`
- 继续复用 `KongqiaoConstants.DYNAMIC_MAX_ROWS = 4`
- 当前的动态 0~4 行，改解释为**修为追加 0~4 行**

最终总行数：

`totalRows = clamp(baseRowsByAptitude + bonusRowsByZhenyuan, 0, KongqiaoConstants.MAX_ROWS)`

这样可以和当前 `MAX_ROWS = 9` 平滑对接：

- 残缺资质：1 + 0~4 = 1~5 行
- 中等资质：3 + 0~4 = 3~7 行
- 绝品资质：5 + 0~4 = 5~9 行

这比“所有人最多只差真元，不差资质”更符合世界观。

## 5. 压力模型（Pressure）

## 5.1 压力的组成

空窍压力由四部分组成：

1. **驻留压力**
   - 蛊虫放进空窍就会占用的常驻负担。
2. **被动压力**
   - 被动用途开启后，持续追加的维持负担。
3. **轮盘预载压力**
   - 主动用途或主动杀招被放进轮盘后，占少量“待发”预算。
4. **爆发与疲劳压力**
   - 主动激活、杀招施放、拆分/炼化/推演后产生的瞬时应力与衰减债务。

## 5.2 压力上限

压力上限不建议与格子数完全绑定，而是由三部分共同决定：

`pressureCap = aptitudeBasePressure + bonusRows * 6 + apertureRankBonus`

其中：

- `aptitudeBasePressure`：体现先天资质差异
- `bonusRows * 6`：体现修为带来的额外承压能力
- `apertureRankBonus`：体现低转 / 高转空窍本体的层次差

建议基础值：

| 资质档位 | 基础压力上限 |
|---|---:|
| 残缺 | 18 |
| 下等 | 28 |
| 中等 | 40 |
| 上等 | 56 |
| 绝品 | 72 |

> `apertureRankBonus` 当前仓库没有现成桥接点，和资质一样，必须通过新的桥接层显式提供，不能在 UI 或文案层硬猜。

## 5.3 单个蛊虫的驻留压力

每个槽位里的蛊虫都会先产生一个基础驻留值：

`residentPressure = guBasePressure × realmGapMultiplier`

建议：

- 普通驻留：`guBasePressure = 1`
- 重型蛊 / 明显强力蛊：可在后续数据中把 `guBasePressure` 提到 `2` 或 `3`

跨转数倍率：

- `realmGap = max(0, guRank - apertureRank)`
- `realmGapMultiplier = 1 + realmGap`

也就是说：

| 情况 | 倍率 |
|---|---:|
| 蛊虫转数 ≤ 空窍转数 | ×1 |
| 高 1 转 | ×2 |
| 高 2 转 | ×3 |
| 高 3 转 | ×4 |

这样能把“低转强养高转蛊”的压迫感直接打出来。

## 5.4 被动用途的持续压力

对 `NianTouData.Usage` 来说，仓库里已经有：

- `cost_duration`
- `cost_total_niantou`

这些字段不能只当展示文本，要转成持续负担。

建议公式：

`passivePressure = (1 + ceil(costDuration / 200) + ceil(costTotalNiantou / 50)) × realmGapMultiplier`

解释：

- 常驻被动至少要吃 1 点持续压力；
- 持续时间长、念头总耗高的效果，本身就应该更“压窍”；
- 高转蛊再乘跨转数倍率。

只有 `TweakConfig` 中被允许生效的被动，才计算这部分压力。

## 5.5 轮盘预载压力

只要一个主动用途 / 主动杀招被放进轮盘，就不应完全零成本。

建议：

- 每个轮盘主动用途：`+1 × realmGapMultiplier`
- 每个轮盘主动杀招：`+2 × realmGapMultiplier`

原因很简单：

- 轮盘不是纯 UI 收藏夹；
- 轮盘表示“你随时准备把这股力量拽出来”；
- 这种待发状态就应该先吃一小部分空窍稳定度。

---

## 6. 压力状态分级

令：

- `effectivePressure = resident + passive + wheelReserve + activationBurst + fatigueDebt`
- `ratio = effectivePressure / pressureCap`

建议分为五档：

| 档位 | 比例 | 含义 | 效果 |
|---|---:|---|---|
| 稳定 | ≤ 0.80 | 运转平稳 | 无额外惩罚 |
| 紧绷 | 0.81 ~ 1.00 | 已逼近极限 | UI 黄警，提示谨慎激活 |
| 超压 | 1.01 ~ 1.20 | 空窍开始不稳 | 禁止新增轮盘技能，部分高耗被动自动熄火 |
| 失控 | 1.21 ~ 1.50 | 已明显反噬 | 主动激活失败率上升 / 直接拒绝，真元恢复下降 |
| 崩窍边缘 | > 1.50 | 危险极高 | 强制封槽、掉疲劳债、持续伤害或念头流失 |

设计重点：

- 不要一上来就“塞满就爆炸”；
- 应该先让玩家感到“还能硬撑，但越来越痛苦”；
- 真正符合蛊真人味道的，是**为了多挂一只高转蛊，长期处在危险边缘**。

## 7. 数据所有权与实现落点

## 7.1 权威状态归属

建议继续让 `KongqiaoData` 作为空窍权威附件，但在其中加入一个新的子状态对象，例如：

- `PressureState`

建议最少包含：

- `burstPressure`
- `fatigueDebt`
- `overloadTier`
- `lastDecayGameTime`
- `lastStablePressure`

其中：

- **常驻驻留压力 / 被动压力 / 轮盘预载压力**：每 tick 可从背包与配置重新推导
- **爆发压力 / 疲劳债务**：需要落盘，否则重登就白洗惩罚

## 7.2 代码锚点

| 目标 | 当前锚点 | 后续动作 |
|---|---|---|
| 容量同步 | `KongqiaoCapacityService` | 从“纯真元映射”改成“资质基础 + 真元追加” |
| 持久化 | `KongqiaoData` | 增加 `PressureState` 并参与 NBT |
| 被动运行 | `GuRunningService` | 在 `onTick/onSecond` 前做压力校验与自动失活 |
| 主动杀招 | `ShazhaoActiveService` | 激活前检查爆发压力，并写入 burst debt |
| 主动蛊用途 | `ServerboundSkillWheelSelectPayload` + `GuRunningService.activateEffectWithResult` | 增加 `PRESSURE_LIMIT` 失败原因 |
| UI 展示 | `KongqiaoScreen` / `TweakScreen` | 增加压力条、负担分解、超压提示 |

## 8. UI 呈现要求

## 8.1 `KongqiaoScreen`

当前空窍界面已经有主面板和按钮区，后续应新增：

- 一条**压力总条**（当前值 / 上限）
- 三条细分文本：
  - 驻留负担
  - 被动负担
  - 爆发 / 疲劳负担
- 当前资质档位
- 当前空窍转数 / 驾驭评价

## 8.2 `TweakScreen`

每个被动用途、每个轮盘主动，都应额外显示：

- 本用途压力消耗
- 跨转数倍率提示
- 当前是否因为超压被强制熄火

## 8.3 激活失败反馈

对主动用途和主动杀招来说，服务端必须返回明确失败语义：

- 条件不足
- 未解锁
- 未实现
- **压力不足 / 空窍超压**

不能继续使用“玩家点了没反应”的表现。

## 9. 实施顺序建议

### 阶段 1：先补权威状态，不碰平衡

1. 在 `KongqiaoData` 内加入 `PressureState`
2. 在 `KongqiaoCapacityService` 中拆出“资质基础 / 真元追加”结构
3. 先做只读 UI 展示，不做惩罚

### 阶段 2：接入运行时校验

1. `GuRunningService` 计算常驻压力
2. `TweakConfig` 与被动开关联动
3. 超压时先只做黄 / 红警告与强制熄火

### 阶段 3：接入主动爆发与疲劳债务

1. `ShazhaoActiveService` 写入 burst pressure
2. 主动蛊用途写入 activation burst
3. 炼化 / 拆分 / 推演路径写入 fatigue debt

### 阶段 4：接入封槽与恢复机制

1. 崩窍边缘时锁定部分槽位
2. 锁定状态随时间 / 修养缓慢恢复
3. 恢复过程走明确提示，不做静默恢复

---

## 10. 测试要求

至少补以下验证：

1. **容量同步测试**
   - 同资质不同真元 → 追加行数不同
   - 同真元不同资质 → 基础行数不同

2. **跨转数压力测试**
   - 高 1 转 / 高 2 转 / 高 3 转蛊进入同一空窍，压力应阶梯上升

3. **被动持续负担测试**
   - 关闭被动后，持续压力下降
   - 开启高耗被动后，进入黄 / 红警告

4. **主动爆发测试**
   - 主动杀招成功施放后，burst pressure 立刻上升
   - 超压时主动触发被明确拒绝

5. **持久化测试**
   - relog / restart 后，fatigue debt 和超压状态不能白洗

---

## 11. 一句话结论

空窍后续不应该再只是“可拓展的背包”。

正式规则应当是：

- **资质决定你能开多大的窍**；
- **修为决定你能把窍养到多厚**；
- **压力决定你是否真的扛得住这些蛊**。

只有把这三层分开，空窍系统才会真正有蛊真人该有的运营压力。
