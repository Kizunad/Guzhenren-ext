# 空窍运转限制与损耗规则（pressure.md 配套）

## 1. 这份文档解决什么问题

`pressure.md` 负责定义“容量”和“压力”的总框架；
本文件负责把“损耗 / 限制 / 失败线”说清楚。

换句话说：

- `pressure.md` 回答：**空窍为什么会有压力**。
- `limit.md` 回答：**压力满了以后会发生什么，以及哪些行为必须付代价**。

---

## 2. 损耗分层

空窍的损耗不能只做成一种数字，否则很快就会变成“纯加减法”。

建议拆成三层：

### 2.1 常驻维持损耗

来源：

- 蛊虫长期驻留
- 被动持续开启
- 主动技能长期挂在轮盘待发

特点：

- 增长慢
- 长时间稳定存在
- 适合做空窍运营的“底噪成本”

### 2.2 爆发性施放损耗

来源：

- 主动蛊用途触发
- 主动杀招施放
- 高转蛊的瞬时催动

特点：

- 立刻抬高压力
- 立刻影响后续操作空间
- 适合做战斗中“连发代价”

### 2.3 结构性疲劳损耗

来源：

- 炼化 / 拆分 / 推演 / 炼蛊重组
- 强行在超压状态下持续运转
- 升仙失败后的空窍虚弱期

特点：

- 不会立刻消失
- 会在数十秒到数分钟内逐渐回落
- 用来表现“这次你虽然没死，但窍已经伤了”

---

## 3. 被动常驻的正式限制

## 3.1 被动不是开关收藏夹

现在的 `TweakConfig` 只表达“开 / 关”，不表达“你是否养得起”。

后续规则必须改成：

- 被动可以开；
- 但开启后会持续吃压力；
- 压力不够时，不是继续白跑，而是进入失稳判定。

## 3.2 被动失稳顺序

当 `effectivePressure > pressureCap` 时，不建议立刻全关，而是按顺序熄火：

1. 先熄火**未加入核心玩法循环的次要被动**
2. 再熄火**压力成本最高的被动**
3. 最后才影响最核心的保命 / 主修被动

实现上建议给每个被动增加一个 `stabilityPriority`（可由后续 metadata / registry 定义）：

- `CORE`
- `IMPORTANT`
- `OPTIONAL`

默认策略：

- `OPTIONAL` 先掉
- 同优先级内，压力越高的先掉

## 3.3 被动强制熄火后的恢复

被动被空窍强制熄火后，不应立即自动恢复，建议加一个短暂冷静期：

- `forcedCooldownTicks = 200 ~ 600`

这样玩家不能靠“刚超压又立刻自动开回去”绕规则。

---

## 4. 主动用途与杀招的爆发损耗

## 4.1 主动用途（蛊虫主动）

对 `GuRunningService.activateEffectWithResult(...)` 这条路径，后续要新增一类失败原因：

- `PRESSURE_LIMIT`

判定顺序建议：

1. usage 是否存在
2. 是否已解锁
3. 是否满足用途条件
4. **本次激活后是否会让压力越过可允许上限**

如果第四步失败：

- 不执行效果
- 返回明确提示
- 不扣其它资源

## 4.2 主动杀招

对 `ShazhaoActiveService.activate(...)` 这条路径，后续要补：

- 激活前压力校验
- 激活成功后的 `burstPressure` 写入
- 激活失败后的明确原因返回

建议杀招爆发损耗的基础值明显高于普通主动蛊用途。

原因：

- 杀招本来就不是普通按键技能；
- 杀招应代表一次完整组织后的爆发；
- 它理应让空窍在短时间内更难连续出手。

## 4.3 轮盘预载限制

轮盘不是无限武器库。

建议：

- 当空窍处于“超压”以上时，禁止继续把新主动用途加入轮盘；
- 当空窍处于“失控”以上时，保留轮盘列表，但禁止触发新增高耗技能；
- 当空窍处于“崩窍边缘”时，只允许触发少数保命级主动。

这能保证“轮盘的丰富度”本身也是经营结果，不是纯 UI 收藏。

---

## 5. 拆分 / 推演 / 炼化的疲劳债务

## 5.1 为什么要有这条线

如果空窍只对“战斗中放技能”收费，而对“拆分、推演、炼化、改造”完全免费，玩法会变成：

- 平时随便折腾空窍结构；
- 真正出手时再承担代价。

这和蛊真人的味道不一致。

更合理的表达是：

- 你每次折腾空窍结构，都会留下“窍内疲劳”；
- 短时间内多次推演 / 拆分 / 炼化，会让空窍越来越不稳；
- 这样玩家必须决定“现在是备战，还是继续做结构实验”。

## 5.2 适用操作

建议纳入疲劳债务的操作：

- 蛊虫拆分
- 用途重整 / 结构重构
- 杀招推演
- 炼蛊 / 合成中的空窍深度参与步骤

## 5.3 疲劳债务规则

建议：

- 每次结构性操作写入固定 `fatigueDebt`
- 高转操作、复杂操作、连续操作叠加更高债务
- `fatigueDebt` 只缓慢衰减，不在操作完成瞬间归零

可实现为：

- 轻度操作：`+4`
- 中度操作：`+8`
- 高危操作：`+12 ~ +20`

衰减方式建议：

- 每秒衰减 1 点
- 或每 40 tick 衰减 1 点

具体数值可以后续调，但“慢恢复”这一点不应取消。

---

## 6. 超压后的正式惩罚线

## 6.1 不同档位的惩罚重点

### 紧绷

- 只警告
- 不直接砍玩法

### 超压

- 不允许新增轮盘技能
- 次要被动自动熄火
- 压力条持续闪黄

### 失控

- 主动高耗技能直接拒绝
- 真元恢复下降
- 被动重启需要等待冷却

### 崩窍边缘

- 封锁部分槽位
- 触发负面状态（如念头流失 / 持续伤害 / 真元滞涩）
- 强制进入恢复期

## 6.2 封槽规则

封槽不是永久损坏，而是“这段时间你的窍扛不住了”。

建议：

- 优先封锁最高压力的若干槽位
- 如果玩家背包有空位，则优先把被顶出来的蛊返还给玩家
- 如果玩家背包满，则走掉落或临时锁定状态

这条规则可以直接复用 `KongqiaoCapacityService` 当前已有的“溢出物品回收”思路。

## 7. 与现有代码的对接要求

## 7.1 `GuRunningService`

必须补：

- 被动执行前的压力校验
- 强制熄火逻辑
- 冷静期逻辑

当前源码中那句：

> `TODO: 可以在这里添加真元消耗判定`

后续应升级为：

- 真元 / 念头 / 压力 三重校验，而不是真元单校验。

## 7.2 `ShazhaoActiveService`

必须补：

- `ActivationFailureReason.PRESSURE_LIMIT`
- 成功激活后写 burst pressure
- 失败后向 UI 返回明确原因

## 7.3 `ServerboundSkillWheelSelectPayload`

必须补：

- 收到 `PRESSURE_LIMIT` 时，给出明确提示文案
- 如果处于崩窍边缘，只允许保命类主动通过

## 7.4 `TweakConfig`

目前 `TweakConfig` 只是偏好设置。

后续不要把真实运行状态塞进 `TweakConfig`。

正确分工应是：

- `TweakConfig`：玩家想怎么配
- `PressureState` / `KongqiaoData`：当前实际上能不能这么跑

---

## 8. UI 呈现补充

除了总压力条，建议在空窍相关 UI 里明确展示：

- 当前档位：稳定 / 紧绷 / 超压 / 失控 / 崩窍边缘
- 最近一次超压原因：
  - 高转蛊压制
  - 被动过多
  - 杀招爆发
  - 拆分 / 推演疲劳
- 恢复预估：多久后能回到稳定区间

只要这些提示缺失，玩家就会觉得系统在“暗中卡我”。

---

## 9. 最小实现顺序

建议严格按下面顺序推进，不要一口气把所有惩罚全开：

1. **先做显示，不做惩罚**
2. **再做被动持续负担**
3. **再做主动爆发负担**
4. **最后做封槽和恢复线**

原因：

- 没有可视化前先上惩罚，调试会非常痛苦；
- 没有持续负担时，爆发负担的平衡没基准；
- 没有爆发负担时，封槽又会显得很突兀。

---

## 10. 一句话落地标准

当这套规则真正生效后，玩家应该感受到的不是：

> “我多了几个格子，多了几个被动。”

而是：

> “我的空窍底盘有限，想强养高转蛊、想常驻太多被动、想连续开杀招，都必须拿稳定度和后续安全去换。”

只有这样，空窍才会从“系统容器”变成“修炼器官”。
