# 空窍全流程玩法审阅版（v1）

## 1. 文档目的

这份文档不是单点机制说明，而是把空窍系统整理成一条**完整玩家生命周期**，用于回答三个问题：

1. 玩家什么时候真正进入空窍玩法？
2. 玩家进入后，完整流程应该怎么走？
3. 后续实现时，哪些地方已经有框架，哪些地方还没有真正落地？

这份文档优先服务于**玩法审阅**，不是代码细节索引。

---

## 2. 空窍在整套玩法里的定位

空窍不应该只是一个“多出来的背包”，而应该是玩家进入蛊道玩法之后的**长期运营核心器官**。

后续完整定位建议如下：

- **前置定位**：空窍是承载蛊虫、念头用途、杀招、轮盘和部分炼化行为的主体。
- **中期定位**：空窍是玩家战斗构筑、养蛊方向、资源压力与风险管理的核心。
- **后期定位**：空窍应和升仙、仙窍、势力、恢复、外部压力系统咬合，而不是独立漂浮。

一句话说：

> 空窍是“蛊虫运营的肉身基础”，不是“功能菜单集合”。

---

## 3. 当前代码里已经存在的主链

从仓库现状看，空窍系统已经有相当多基础设施：

### 3.1 数据与持久化

- `KongqiaoOwner`
  - 把玩家 / 其他实体统一抽象成空窍持有者。
- `KongqiaoData`
  - 已承载空窍背包、攻击背包、喂食背包，并支持 NBT。
- `NianTouUnlocks`
  - 已承载念头用途解锁、杀招解锁、鉴定过程、推演提示。
- `TweakConfig`
  - 已承载被动开关、轮盘技能列表。
- `ActivePassives`
  - 已承载当前运行中的被动 usageId 集合。

### 3.2 入口与同步

- `KongqiaoAttachmentEvents`
  - 当前会在玩家进入世界时确保空窍附件存在，并在 clone 时复制。
- `KongqiaoService`
  - 已负责创建空窍背包、打开菜单、扩容、打开攻击背包 / 喂食背包。
- `ServerboundKongqiaoActionPayload`
  - 已负责把客户端空窍操作转到服务端。
- `KongqiaoSyncService`
  - 已做按需同步。
- `ClientboundKongqiaoSyncPayload`
  - 已做客户端状态同步。
- `PacketSyncKongqiaoData`
  - 已做念头 / 杀招数据快照同步，解决多人服描述显示问题。

### 3.3 玩法运行时

- `KongqiaoCapacityService`
  - 已按最大真元同步空窍容量。
- `NianTouMenu` + `NianTouTickHandler`
  - 已支持鉴定启动、按 tick 扣念头、解锁用途。
- `GuRunningService`
  - 已支持被动用途在 tick / second 维度运行。
- `ShazhaoUnlockService`
  - 已支持推演候选筛选和成功率计算。
- `ShazhaoRunningService`
  - 已支持被动杀招按秒运行。
- `ShazhaoActiveService`
  - 已支持主动杀招校验和触发。
- `ServerboundSkillWheelSelectPayload`
  - 已支持轮盘主动触发。

### 3.4 UI 层

- `KongqiaoScreen`
  - 主空窍背包界面。
- `NianTouScreen`
  - 念头鉴定界面。
- `TweakScreen`
  - 调整面板，管理被动和轮盘。
- `SkillWheelScreen`
  - 主动轮盘。

结论很明确：

> 空窍不是“没做”，而是“已经有大量模块，但还缺一份真正完整的玩法主流程和几条关键规则”。

## 4. 当前最大缺口

### 4.1 缺“从哪开始”的正式前置

现在代码里，玩家进世界就会自动挂上空窍附件。

这意味着：

- 技术上可以直接用；
- 但玩法上没有“获得空窍 / 进入空窍玩法”的桥段。

如果后续不补这条线，玩家会觉得空窍只是系统默认功能，而不是修炼进程的一部分。

### 4.2 缺“空窍为什么有代价”的正式规则

当前 `KongqiaoCapacityService` 只看真元同步格子数。

这意味着：

- 容量有了；
- 但压力没有；
- 高转蛊压制没有；
- 被动常驻成本没有；
- 杀招爆发应力没有。

所以空窍目前更像“扩展型构筑背包”，还不像真正的器官系统。

### 4.3 缺“失败与恢复”这条线

现在有鉴定推进，也有被动和杀招运行，但还没有一条完整的：

- 超压
- 失控
- 封槽
- 疲劳债
- 恢复

玩法线。

如果没有这一层，玩家只会不断叠功能，不会真正做取舍。

### 4.4 缺“端到端测试矩阵”

当前能找到的主要是一些 JUnit 测试，空窍主链缺少明确的 GameTest / 集成验证矩阵。

这会导致后续空窍一旦加重规则，很容易出现：

- relog 状态丢失
- 被动和 UI 不一致
- 轮盘触发与服务端状态不同步
- 鉴定 / 推演进度在多人服表现异常

## 5. 建议的空窍完整玩家流程

下面是后续建议的**正式全流程**。

## 5.1 阶段 A：进入空窍玩法

### 当前现实

技术上，玩家进入世界就有空窍附件。

### 后续正式表达

建议区分：

- **技术附件存在**
- **玩法正式启用**

后续应至少加一个“空窍启用状态”，使玩家经历：

1. 拥有空窍基础载体
2. 通过前置流程正式启用空窍玩法
3. 启用后才开放完整空窍 UI / 构筑 / 压力系统

这样可以和后面的升仙、资质、势力都更好联动。

## 5.2 阶段 B：容量建立

玩家正式进入空窍玩法后，第一感知不应是“我有几个槽”，而应是：

- 我的空窍底盘如何
- 当前能装多少蛊
- 为什么别人能比我装更多

后续应把容量正式拆成：

- 资质基础容量
- 真元 / 修为后天扩容

这一步是整个空窍运营的底盘。

## 5.3 阶段 C：念头鉴定

这是当前仓库里已经比较清楚的一条链：

1. 把物品放进 `NianTouMenu`
2. 启动鉴定
3. `NianTouTickHandler` 按 tick 扣念头并推进
4. 完成后在 `NianTouUnlocks` 中记录 usage 解锁

后续重点不是重做，而是补足：

- 鉴定前置条件
- 鉴定失败 / 暂停反馈
- 与压力 / 疲劳 / 资质的联动

## 5.4 阶段 D：空窍驻留与被动运行

玩家把蛊虫放进空窍后，系统进入真正的“养窍”状态：

1. 蛊虫驻留产生常驻负担
2. 已解锁用途进入可运行集合
3. `GuRunningService` 每 tick / 每秒运行被动
4. `ActivePassives` 记录当前活跃被动

这是空窍玩法的常态主循环。

后续重点要补：

- 被动持续代价
- 超压时的自动熄火顺序
- 被动恢复冷静期

## 5.5 阶段 E：杀招推演与被动杀招

当前也已有明确链路：

1. 玩家在 `NianTouScreen` 尝试推演杀招
2. `ShazhaoUnlockService` 根据已解锁蛊虫用途判断候选和概率
3. 解锁后写入 `NianTouUnlocks`
4. `ShazhaoRunningService` 驱动被动杀招

后续要补的是：

- 推演操作的疲劳债
- 杀招之间的定位分层
- 被动杀招与普通被动 usage 的优先级冲突

## 5.6 阶段 F：主动轮盘与主动出手

当前链路也已经成形：

1. `TweakScreen` 配置轮盘技能
2. `SkillWheelScreen` 展示轮盘
3. `ServerboundSkillWheelSelectPayload` 发送选择
4. 普通主动用途走 `GuRunningService.activateEffectWithResult`
5. 主动杀招走 `ShazhaoActiveService.activate`

这条线后续最关键要补：

- 轮盘预载成本
- 爆发压力
- 压力不足时明确失败反馈
- 连续施放后的疲劳债

## 5.7 阶段 G：超压、失控与恢复

这是当前最缺的一段，也是后续必须补上的整条线：

1. 驻留压力过高
2. 被动维持过高
3. 主动爆发把空窍推过上限
4. 进入紧绷 / 超压 / 失控 / 崩窍边缘
5. 触发被动熄火、主动拒绝、封槽、恢复债务
6. 玩家通过时间、修养、资源、特殊操作把空窍拉回稳定区

没有这条线，空窍就没有真正的运营代价。

## 5.8 阶段 H：多人同步与投影一致性

空窍的数据和 UI 很依赖同步，这点仓库里已经有基础解决方案：

- `KongqiaoSyncService`
- `ClientboundKongqiaoSyncPayload`
- `PacketSyncKongqiaoData`

后续所有新增规则——尤其是压力、超压状态、封槽、疲劳债——都必须走同一套服务端权威同步链，而不是让 UI 自己猜。

## 6. 现在最应该明确的正式状态分层

建议后续把空窍状态分成五层：

1. **启用状态**
   - 玩家是否正式进入空窍玩法。
2. **容量状态**
   - 当前总格子 / 基础格子 / 后天扩容。
3. **解锁状态**
   - 念头用途、杀招、轮盘配置。
4. **运行状态**
   - 当前活跃被动、被动熄火、主动待发。
5. **稳定状态**
   - 当前压力、疲劳债、超压等级、封槽状态。

只要这五层还没分开，后续实现就容易把：

- UI 偏好
- 临时运行态
- 永久解锁态
- 压力惩罚态

混在一起。

## 7. 当前代码里“已经够用的部分”

这些部分不建议重做：

### 7.1 鉴定主链

- `NianTouMenu`
- `NianTouTickHandler`
- `NianTouUnlocks`

这条链已经够清楚，应在其上叠规则，而不是推翻。

### 7.2 数据快照同步

- `PacketSyncKongqiaoData`

这条链已经明确解决了专用服务器环境下客户端看不到数据的问题，后续应继续复用。

### 7.3 主动轮盘基本结构

- `TweakConfig`
- `SkillWheelScreen`
- `ServerboundSkillWheelSelectPayload`

这条链已经足够承担后续“压力限制 + 失败提示 + 高耗拒绝”的新增规则。

## 10. 一句话总结

空窍当前不是“没有内容”，而是：

> 已经有了大量能运行的模块，但还缺一份把进入、承载、鉴定、驻留、轮盘、杀招、超压、恢复、同步、测试全部串成一条主线的总蓝图。

后续只要把这条主线补齐，空窍就能从“功能集合”真正升级成“可长期经营、可承担风险、可承接升仙前后玩法”的核心系统。
